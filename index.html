          The prompt is the product's brain.

          The earnings review is only as good as the instructions we hand the model. Here's how we frame that instruction file, what changed from v1 to v2, and the quality it produces on real data.

          How we frame an instruction file — the 5 layers

          Every doc that drives the model is built top-to-bottom in these layers. The point: a rule lives in exactly one place, every output has a checkable shape, and quality improves by editing the doc — not re-prompting from scratch.

          1. 1
            Purpose & audience
            One line: who reads it, what they produce, at what level. "Hedge-fund-grade teardown; assume an expert; skip basics." This calibrates everything below.
          2. 2
            Operating protocol
            How to execute, not what. "One task at a time; reset context between tasks; validate before moving on."
          3. 3
            Global conventions — the single source of truth
            Every rule more than one task needs (sources, number formatting, table rules, reusable procedures) stated once and numbered, so tasks reference it (per §1.3) and never restate. The highest-leverage layer.
          4. 4
            Tasks, each with a contract
            Objective → Sources (with priority) → field logic → Output Contract (the exact table template) → pre-flight check. Each task inherits §3 and states only what's unique.
          5. 5
            Reference data, in its own file
            Lookups too big to inline (our master_kpi.md — ~6,600 KPI definitions keyed by ticker) live separately, so the instruction file stays readable.

          Plus two closers: a failure-modes list (negative examples catch the errors rules miss), and a golden example — one worked, verified output. The example is earned, never invented: a faked example teaches the model the wrong target, so it comes from a real output a human has marked up and blessed.

          v1 → v2: what changed, and why it matters

          v1 — the original
          • The style guide & formatting rules were copy-pasted into every task — the same blocks appeared 15 times.
          • Change one formatting rule → you had to find and fix it in 15 places. Miss one and the output drifts.
          • Rules described in prose; no checkable output shape.
          • No failure-mode list; no graceful handling when data is missing.
          v2 — the rewrite
          • All shared rules collapsed into one §1 "Global Conventions" — referenced everywhere, stated once (the repeated blocks: 15 → 2).
          • Change a rule in one place → it applies to all 8 tasks. Drift becomes structurally impossible.
          • Every task ends in an exact output contract + a pre-flight check ("every row has exactly 6 pipes").
          • Added a failure-modes section, graceful KPI lookup, and Task 8 (the Drivers loop).

          The win isn't fewer lines (303 vs 308 — about the same). It's that the new file fits more — 8 tasks, failure modes, graceful fallbacks — into the same space, because nothing is repeated. One source of truth, referenced, instead of fifteen copies that drift.

          In plain English — what I changed and why

          • Stopped repeating the rules. The formatting/style guide used to be pasted into every task (15 copies). I pulled it into one numbered section the tasks point at. Now a rule can't drift, and editing it is a one-line change.
          • Gave every task a contract. Each task ends with the exact table it must output and a quick self-check. The model has a target to hit, not a vibe to interpret.
          • Fixed a real mismatch. The prompt's P&L table said 5 columns, but the actual output had added a 6th ("vs Consensus"). I made the rule explicit so prompt and output agree.
          • Made missing data fail gracefully. If a company isn't in our KPI dictionary, the model now pulls the KPIs the company actually reported and flags it — instead of inventing metrics.
          • Added a failure-modes list. Concrete "do NOT do this" examples (don't write N/A, don't invent guidance, don't pad columns) — these catch the mistakes that positive rules miss.
          • Added Task 8 — the Drivers loop. The part that's ours: analyze the quarter through our saved debates, and if we have no thesis on file yet, skip it and say so rather than fake one. This is what turns a great earnings teardown into our thinking, tracked quarter over quarter.
          • Left the example blank — on purpose. The single best quality lever is a worked example, but a wrong one teaches the wrong thing. So it stays empty until you and Pankaj mark up a real output and bless it. Then it goes in.

          Bottom line: the v2 prompt is the same length but does more, can't drift, checks its own output, handles missing data honestly, and reaches toward the Drivers loop. Quality now compounds in the document — every quarter's mistakes become next quarter's guardrails.
