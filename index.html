          The prompt is the product's brain.

          The earnings review is only as good as the instructions we hand the model. Here's how we frame that instruction file, how it evolved (v1 → v2 → v3), and the quality it produces on real data.

          How we frame an instruction file — the 5 layers

          Every doc that drives the model is built top-to-bottom in these layers. The point: a rule lives in exactly one place, every output has a checkable shape, and quality improves by editing the doc — not re-prompting from scratch.

          1. 1
            Purpose & audience
            One line: who reads it, what they produce, at what level. "Hedge-fund-grade teardown; assume an expert; skip basics." This calibrates everything below.
          2. 2
            Operating protocol
            How to execute, not what. "One task at a time; reset context between tasks; validate before moving on."
          3. 3
            Global conventions — the single source of truth
            Every rule more than one task needs (sources, number formatting, table rules, reusable procedures) stated once and numbered, so tasks reference it (per §1.3) and never restate. The highest-leverage layer.
          4. 4
            Tasks, each with a contract
            Objective → Sources (with priority) → field logic → Output Contract (the exact table template) → pre-flight check. Each task inherits §3 and states only what's unique.
          5. 5
            Reference data, in its own file
            Lookups too big to inline (our master_kpi.md — ~6,600 KPI definitions keyed by ticker) live separately, so the instruction file stays readable.

          Plus two closers: a failure-modes list (negative examples catch the errors rules miss), and a golden example — one worked output that anchors the model on the target shape. The example is earned, never invented: a faked example teaches the wrong target. v3 seeds it from a real Gemini run, flagged DRAFT — UNBLESSED until Pankaj marks it up and blesses it — honest about its status, not faked.

          v1 → v2 → v3: two different kinds of improvement

          v2 fixed how the file is built (no drift). v3 fixed what the file asks for — because the best real output was already richer than the prompt that produced it.

          v1 — the original
          • The style guide was copy-pasted into every task — the same blocks appeared 15 times.
          • Change one rule → fix it in 15 places. Miss one and the output drifts.
          • Rules in prose; no checkable output shape; no failure-mode list.
          v2 — made it DRY
          • Shared rules collapsed into one §1 "Global Conventions", stated once (15 → 2 copies).
          • Change a rule in one place → applies to all tasks. Drift becomes structurally impossible.
          • Every task got an output contract + pre-flight, a failure-modes list, and Task 8 (the Drivers loop).
          v3 — made it richer
          • The contract now demands the analyst's Read (the so-what) on every task — the biggest data-dump-→-analyst lever.
          • Added a "vs Consensus" column, inline source tags, an Executive Summary, cross-task reconciliation, and a not-found rule (never invent a number).
          • Seeded a real golden example (flagged DRAFT until blessed), and a rule to adapt the P&L rows to banks / REITs / restaurants.

          v2's win was structural: one source of truth instead of fifteen copies that drift. v3's win was the harder one — the blessed outputs were already doing things the prompt never asked for (a Read, a consensus bar, source tags). v3 writes those proven behaviors back into the contract, so the prompt finally specifies the quality the best output already had — instead of relying on the model's instinct to exceed it.

          In plain English — what v3 changed and why

          v2 fixed the plumbing (no repetition, no drift). These are the v3 changes — the ones that raise the quality of the report itself, all earned from things the best real outputs were already doing.

          • Every task now ends with a Read. One line of judgment — the so-what, not a restatement of the numbers. "A quality beat on a soft top line" tells you more than ten rows of data. This is the single biggest step from data-dump to analyst's read.
          • Added a "vs Consensus" column — with a hard rule against guessing. The print sits next to the Street bar so you see the beat/miss immediately. If we don't have a consensus number in the corpus, the cell says pending — never an invented figure.
          • Every load-bearing number carries a source tag. [SE-PR] = press release, [SE-CT] = call, [SE-SS] = sell-side. You can trace any claim back to where it came from — and a missing number reads N/F ("not found"), never a fabricated one.
          • Added an Executive Summary — "The View." The part a partner reads first: what the quarter was, the three things that matter, and the risks to watch. Written last (so it reconciles to the tables), shown first.
          • Made the report check itself. The same metric must read identically across every task; the model recomputes the math instead of trusting a source-stated growth rate; and if a whole section is missing data, it says so honestly rather than papering over it.
          • Taught it to adapt to the business. A bank has no gross margin; a REIT reports FFO. v3 tells the model to swap the P&L rows for what the business model actually uses, instead of forcing a one-size template.
          • Seeded a real example — flagged honestly. The single best quality lever is a worked example. v3 now includes two real ones (CAVA + Avery), marked DRAFT — UNBLESSED. They anchor the model on the right shape now; once you and Pankaj mark up a real run, the blessed version replaces the draft and the flag comes off. Honest about its status, never faked.

          Bottom line: v2 made the prompt impossible to break; v3 made the output better — a mandatory analyst's Read, a verified consensus bar, traceable sources, an exec summary, and a self-check. The improvement is that the prompt finally demands the quality the best output already had — instead of hoping the model exceeds the spec.
